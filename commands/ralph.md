# Ralph Wrapper Skill

유저의 자연어 요청을 구조화된 Ralph Loop 프롬프트로 변환한다.

## 입력 파싱

유저 메시지에서 다음을 추출한다:
- **반복 횟수**: "N번", "N회", "N iterations", "N번만" → `--max-iterations N` (없으면 기본 3)
- **작업 내용**: 나머지 전체 텍스트

## 구조화 규칙

아래 템플릿으로 프롬프트를 작성한다.
작업 내용이 모호하더라도 최대한 의도를 추론해서 채운다. 추측이 불가능한 항목만 `[미정]`으로 남긴다.

### 핵심 원칙: 핵심 프롬프트는 짧고 자율적으로

세부 설계를 프롬프트에 인라인하면 모델이 그 설계 박스를 못 벗어난다. 대신 **목적·커버리지를 스스로 파악하게 시키고**, 비판적 검토 마인드를 강하게 심는다. 계획 문서가 있으면 "참고"로만 두고 빠진 것은 자율적으로 채우게 한다.

작성 규칙:
1. **목적/대상만 명확히** — "무엇을 달성해야 하는지"를 한두 문장으로. 설계·구조·새 키워드 나열 금지.
2. **계획 문서는 SoT가 아닌 참고** — "X 문서를 읽고 목적과 커버리지를 파악한 뒤 진행해라" 톤. "X대로 따라하라" 금지.
3. **자율 확장 권한 부여** — "부족한 것/설계 누락은 알아서 더 넣어라", "관련 문서·테스트·훅·인프라 모두 점검해라"를 명시.
4. **비판적 자기검토 마인드** — "회차가 지날수록 비판적 사고로 이전 회차 산출물을 면밀히 검토해라", "못된 직장 상사의 마음으로 평가해서 개선점을 도출해라"를 명시.
5. 그 외 컨텍스트 관리·최소 2회차·검증 루틴·완료 조건은 아래 스캐폴드 그대로 유지.

```
## 목표
[1~3문장. 무엇을 달성해야 하는지만. 설계·구조·키워드 나열 금지.
계획/참고 문서가 있다면: "<문서 절대경로>를 읽고 목적과 커버리지를 명확히 파악한 뒤 진행해라.
문서에 적힌 내용은 참고만 하고 목적을 달성하기 위한 너의 모든 노력을 기울여라.
구현하다가 부족한 것이 있거나 설계에 누락된 것이 있다면 알아서 더 넣고 관련된 문서·테스트·훅 등의 모든 인프라를 점검해라.
회차가 지나갈수록 비판적인 사고로 이전 회차에서 작업한 게 문제가 있는지 면밀하게 검토하면서 진행하고,
아주 못된 직장 상사의 마음으로 평가해서 개선점을 도출해라."
이 톤을 유지한다. 유저 원문이 있으면 가급적 그대로 사용.]

## 컨텍스트 관리
매 회차 시작 전 `/clear`로 컨텍스트를 클리어한다.
따라서:
- 매 회차 시작 시 progress 파일(`<PROGRESS_PATH>`)을 읽어 현재 상태 파악
- 이전 회차 기억에 의존하지 않는다. 파일을 직접 읽어 "이미 끝난 것"과 "아직 남은 것"을 판별
- 매 회차 종료 시 progress 파일에 완료 항목과 남은 항목을 기록
- `<PROGRESS_PATH>`는 3단계에서 세션 스코프 절대경로로 치환된다 — `/tmp` 사용 금지, 다른 세션과 교차오염 없음.

## 최소 2회차 강제 규칙
- **1회차**: 작업 수행 (구현/수정). 1회차에서는 절대 완료 선언하지 않는다.
- **2회차 이후**: 이전 회차 결과를 검증. 검증 통과 시에만 완료 가능.
- progress 파일에 `iteration: N` 기록. N < 2이면 완료 선언 금지.

## 매 회차 종료 시 검증 루틴
1. **잔여 참조 grep**: `grep -r "[옛 키워드들]" [대상 경로]`로 변경 전 이름이 남아있는지 스캔
2. **자가 질문**:
   - 이번 회차에서 수정하거나 발견한 문제가 있었는가?
     → 있었다면: 다음 회차에서 파급 효과 확인
   - 대상 전체를 다시 훑었을 때 아직 손대지 않은 부분이 남아 있는가?
   - 지금 멈춰도 목표가 충분히 달성됐다고 확신할 수 있는가?

## 적대적 교차 검증 (완료 선언 직전 필수)

"내가 반드시 빵꾸를 냈을 것이다"는 전제로, 아래 6가지를 반드시 실행한다.
**이 검증을 건너뛰고 완료 선언하는 것은 금지.**
**2건 미만 이슈 발견 시 검증이 부족한 것으로 의심하라. 더 파라.**

### A. 코드 블록 실존 검증
수정된 모든 .md 파일의 코드 블록에서 참조하는 함수명, CLI 파라미터, 파일 경로, 클래스명을 **전수** 추출하고,
실제 코드에서 grep해서 존재하는지 확인한다. 1~2개만 샘플링하면 안 된다.
- 예: 스킬 .md에 `executor.py --phase system-design`이 있으면 → executor.py argparse에 `--phase`가 실제로 있는지
- 예: .md에 `run_ux_validation()`이 있으면 → core.py에 def run_ux_validation이 있는지

### B. 주어-위치 교차 검증
문서에서 "X 루프에서 Y를 한다"는 문장의 **주어(X 루프)가 실제 코드 위치와 맞는지** 확인한다.
- 예: "plan 루프에서 architect(SD)를 호출한다" → plan_loop.py에 architect 호출이 있는지
- 예: "설계 루프에서 designer를 호출한다" → 메인 Claude 오케스트레이션인데 harness 코드에 있으면 모순

### C. 문서 간 수치/이름 일치
마커 이름, 타임아웃 값, 에이전트 이름 등이 참조하는 모든 문서에서 동일한지 확인한다.
- 예: orchestration-rules.md의 마커 테이블 vs core.py Marker enum vs 에이전트 .md
- 예: 타임아웃 600s가 orchestration-rules.md와 plan_loop.py에서 동일한지

### D. 엣지케이스 시나리오 워크스루
변경된 코드 경로에 대해 최소 **5개** 엣지케이스를 상상하고 코드를 따라간다.
- 예: "ux-flow.md는 있는데 prd.md가 없으면?" → plan_loop.py에서 어떤 경로를 타는지
- 예: "ux-architect가 마커 없이 끝나면?" → UNKNOWN → UX_FLOW_ESCALATE 처리되는지

### E. 리턴값 호출자 체인 추적 (신규)
새로 추가된 모든 리턴값/마커에 대해, **호출자가 해당 값을 처리하는지** 끝까지 추적한다.
- 함수가 "X"를 리턴 → 호출자가 "X" 분기를 갖고 있는지 grep
- 에이전트가 "MARKER_Y"를 발행 → 하네스/스킬이 "MARKER_Y"를 파싱하는 코드가 있는지
- 문서에 "Z 경로"가 정의 → 스킬/코드가 실제로 Z를 호출하는 구체적 방법이 명시되어 있는지

### F. "변경 시 함께 업데이트" 테이블 전수 대조 (신규)
`orchestration-rules.md`의 "이 파일 변경 시 함께 업데이트할 대상" 테이블을 읽고,
이번에 변경한 파일 각각에 대해 대응 파일이 실제로 업데이트됐는지 확인한다.
- 예: "에이전트 추가" → agent-boundaries.md + CLAUDE.md + 스크립트 전부 갱신됐는지
- 예: "마커 추가" → core.py Marker enum + markers.sh.bak + 해당 에이전트 .md 전부 갱신됐는지

**A~F 중 하나라도 불일치를 발견하면 완료 선언 금지. 수정 후 재검증.**

**완료 조건** (모두 충족해야 함):
- 최소 2회차 이상 진행됨 (progress 파일의 iteration >= 2)
- 세 자가 질문 모두 "문제 없음 / 남은 것 없음 / 확신함"
- 적대적 교차 검증 A~F 전부 통과 (progress 파일에 각 항목 결과 + 발견 건수 기록)
- 위 조건 충족 시 아래를 출력:
<promise>RALPH_DONE</promise>

조건 미충족 시 절대 출력하지 않는다. 루프를 탈출하기 위해 거짓으로 출력하는 것은 엄격히 금지된다.
```

## 실행 절차

### 1단계: 구조화된 프롬프트 생성 후 유저에게 제시

프롬프트를 생성한 뒤 아래 형식으로 유저에게 보여준다:

```
---
**Ralph Loop 실행 설정**

- 최대 반복: N회
- 완료 시 조기 탈출: 활성화 (RALPH_DONE 출력 시)

**구조화된 프롬프트:**

[생성된 프롬프트 전체 출력]

---
이대로 실행할까요? 수정이 필요하면 말씀해 주세요.
```

### 2단계: 유저 확인 대기

- 유저가 "응", "ㅇㅇ", "ok", "고", "실행", "그래" 등 긍정 응답 → 3단계 진행
- 유저가 수정 요청 → 해당 부분만 수정 후 다시 1단계 출력
- 유저가 취소 → 종료

### 3단계: 세션 스코프 경로에 task 저장 후 ralph-loop 호출

`/tmp` 글로벌 경로는 세션 A·B의 ralph-loop가 서로의 transcript/progress를 claim하는 버그를 일으킨다(OMC도 세션별 폴더로 해결). 항상 **현재 세션 스코프 경로**에 저장한다.

1. Bash 도구로 세션 ralph 디렉토리를 확보하고 경로를 추출한다:

```bash
SID="${HARNESS_SESSION_ID:-$(cat ~/.claude/harness-state/.session-id 2>/dev/null)}"
# 세션 ID가 없을 때도 다른 세션과 섞이지 않도록 프로세스 고유 슬롯 사용
# (단일 `_global` 공유 폴백은 원래 버그의 재발 경로라 피한다)
if [ -z "$SID" ]; then
  SID="_pid-$$-$(date +%s)"
fi
RALPH_DIR=~/.claude/harness-state/.sessions/"$SID"/ralph
mkdir -p "$RALPH_DIR"
printf 'TASK_PATH=%s/task.md\nPROGRESS_PATH=%s/progress.md\n' "$RALPH_DIR" "$RALPH_DIR"
```

2. 1단계에서 생성한 프롬프트의 `<PROGRESS_PATH>` 자리표시자를 위에서 출력된 **절대 `PROGRESS_PATH`**로 치환한다. 그 다음 Write 도구로 최종 프롬프트를 **절대 `TASK_PATH`**(예: `/Users/…/.claude/harness-state/.sessions/<SID>/ralph/task.md`)에 저장한다.

3. Skill 도구로 ralph-loop 호출 (짧은 참조 프롬프트 사용):

```
Skill: ralph-loop:ralph-loop
Arguments: "Read <TASK_PATH 절대경로> and execute the task described there" --max-iterations N --completion-promise "RALPH_DONE"
```

4. 청소는 수동으로 하지 않는다. 세션 ralph 디렉토리는 Phase 3 stale cleanup이 6시간 TTL로 자동 정리한다. 같은 세션에서 ralph를 다시 돌리면 `task.md`/`progress.md`가 새 내용으로 덮어써진다.
